# SpecControl AI — Sample Dataset (Phase 0)

Eval-датасет для PoC нормоконтроля технической документации. Используется для замера recall/precision pipeline и определения scope нормативной базы.

## Таксономия ошибок

Phase 0 фокусируется на **2 разнородных категориях**, чтобы продемонстрировать кросс-доменную применимость pipeline:

| # | Категория | Типичные параметры | Ключевые нормативы |
|---|---|---|---|
| 1 | Сечение кабелей vs ток нагрузки | сечение (мм²), тип кабеля, расчётный ток | ПУЭ 7-е изд. (гл. 7.1), ГОСТ Р 50571.5.52-2011 |
| 2 | Воздухообмен в помещениях | расход воздуха (м³/ч), тип помещения, кратность | СП 60.13330.2020, СанПиН 1.2.3685-21 |

Параметры и нарушения, не попадающие в эти 2 категории, в Phase 0 **игнорируются** (не аннотируются, не учитываются в метриках).

## Структура

```
data/samples/
├── README.md              # этот файл
├── schema.json            # JSON Schema для ground_truth.json
└── <doc_id>/
    ├── document.pdf
    └── ground_truth.json
```

`doc_id` — snake_case, например `electrical_spec_001` или `ventilation_office_002`.

## Стратегия сбора (Strategy B — primary)

Реальные PDF с размеченными ошибками не публикуются в открытом доступе. Вместо этого мы строим **синтетический датасет** на базе двух открытых источников:

1. **Открытые учебные спецификации** (корректные baseline PDF) — курсовые, дипломы и методички из репозиториев технических вузов (ВлГУ, ТПУ, ЮУрГУ, chertezhi.ru и т.п.)
2. **Каталоги типовых ошибок от госэкспертиз** — публичные перечни нарушений с привязкой к конкретным пунктам нормативов

Преимущества по сравнению с поиском реальных документов:

- Ground truth получается **детерминированно** (мы сами знаем, что инжектировали)
- Привязка к нормативам точная (из каталогов экспертиз, а не домыслена)
- Можно гарантировать покрытие обеих категорий таксономии

### Шаг 2a — Error catalog

Выписать 10–15 ошибок в `data/samples/error_catalog.md` из следующих источников:

- [СПб госэкспертиза — электроснабжение](https://www.spbexp.ru/docs/tipovye-oshibki/sistema-elektrosnabzheniya/)
- [СПб госэкспертиза — ОВиК](https://www.spbexp.ru/docs/tipovye-oshibki/otoplenie-ventilyatsiya-i-konditsionirovanie-vozdukha/)
- [Приморская госэкспертиза](https://primgosexpert.ru/tipovie_oshibki/) (пп. 72–87 — электрика, пп. по вентиляции)
- [Вологодская госэкспертиза](http://gosexpert35.ru/errors/)

Формат:

```markdown
| ID | Категория | Описание ошибки | Нарушаемый норматив | Источник |
|---|---|---|---|---|
| E01 | cable_section | Сечение кабеля не обосновано расчётом по току | ПУЭ 7.1.34 | spbexp |
```

Скачать 3–5 корректных учебных спецификаций в `data/samples_clean/<doc_id>/document.pdf`.

Кандидаты для электрики:

- [ВлГУ — курсовая Электроснабжение](https://dspace.www1.vlsu.ru/bitstream/123456789/6225/1/01644.pdf)

Для ОВиК — отдельный поиск учебных материалов по вентиляции.

Фильтры:

- Русский язык, PDF с текстовым слоем
- < 20 МБ, < 50 страниц
- Содержит таблицу спецификации или расчётную секцию, куда можно инжектировать ошибку
- Покрытие: минимум 2 электрика + 1 ОВиК (или наоборот)

### Шаг 2c — Инжекция

Для каждого документа (кроме одного чистого):

1. Выбрать 2–3 ошибки из `error_catalog.md`
2. Отредактировать PDF (LibreOffice Draw для ручной правки или `pymupdf` для точечных правок текста)
3. Сохранить модифицированную версию в `data/samples/<doc_id>/document.pdf`
4. Оригинал оставить в `data/samples_clean/<doc_id>/document.pdf`

**Обязательно:** хотя бы один документ остаётся без инжекции (`expected_violations: []`) — чистый кейс для проверки false-positive rate pipeline.

## Правила аннотации (Step 4 плана)

Для каждого документа заполнить `ground_truth.json` строго по `schema.json`:

- `expected_parameters` — все параметры из таксономии, которые есть в документе
- `expected_violations` — только явные нарушения; серые зоны не включать (уронит precision метрик)
- `expected_normatives` — только те нормативы, на которые ссылается хотя бы один violation
- `source_page` — 1-indexed, как показывает PDF viewer
- `source_text_hint` — уникальный фрагмент 30–60 символов для быстрого поиска глазами

**Обязательно:** хотя бы один документ с `expected_violations: []` (чистый кейс для проверки false-positive rate pipeline).

Валидация: `jsonschema -i data/samples/<doc_id>/ground_truth.json data/samples/schema.json`

## Лицензии и PII

- Документы берутся **только из открытых публичных источников** (тендеры, учебные материалы, открытые каталоги)
- Перед коммитом проверить отсутствие реальных PII (ФИО, адреса, телефоны, email). При обнаружении — либо закрасить в PDF, либо заменить документ
- Источник каждого документа фиксируется в `ground_truth.json.source_url`
