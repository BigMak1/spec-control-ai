# Product Proposal: SpecControl AI

## 1. Обоснование идеи

### Прикладная задача

Нормоконтроль технической документации — обязательный этап в строительстве, промышленном проектировании и производстве. Специалист должен вручную сверить десятки параметров в документе с требованиями из нескольких нормативных документов (ГОСТы, СП, СНиПы). На одну проверку уходит от нескольких часов до нескольких дней.

### Почему сейчас

- LLM научились надёжно извлекать структурированные данные из неструктурированного текста
- RAG-подход позволяет работать с актуальной нормативной базой без дообучения модели
- Агентные фреймворки дают возможность декомпозировать сложную задачу на шаги с контролем качества на каждом этапе

### Почему нельзя решить простым чатом

- Нормативные документы не помещаются в контекстное окно — нужна внешняя база знаний (vector store)
- Требуется извлечение структурированных параметров из загруженных документов — нужны инструменты парсинга
- Сопоставление параметров с нормативами требует пошаговой логики с промежуточным состоянием — нужен агент с памятью
- Результат должен быть воспроизводимым и верифицируемым — нужен structured output и трассировка решений

## 2. Цель и метрики успеха

### Цель

Создать PoC агентной системы, которая за < 3 минут извлекает параметры из загруженного PDF/DOCX-документа, находит несоответствия нормативам и формирует структурированный отчёт.

### Продуктовые метрики

| Метрика | Целевое значение |
|---------|-----------------|
| Время обработки одного документа (до 20 стр.) | < 3 мин |
| Доля найденных реальных несоответствий (recall) | ≥ 70% |
| Доля корректных замечаний из всех выданных (precision) | ≥ 80% |
| Доля успешных запусков без ошибок (reliability) | ≥ 90% |

### Агентские метрики

| Метрика | Целевое значение |
|---------|-----------------|
| Среднее количество шагов агента на документ | ≤ 10 |
| Доля зацикливаний (loops) | < 5% |
| Частота использования fallback-логики | < 15% |

### Технические метрики

| Метрика | Целевое значение |
|---------|-----------------|
| p95 latency полного пайплайна | < 180 с |
| Стоимость обработки одного документа (API calls) | < $0.50 |
| Uptime сервиса (в рамках PoC) | > 95% |

## 3. Сценарии использования

### Основной сценарий (happy path)

1. Пользователь загружает PDF-документ (проект электроснабжения жилого дома, 15 страниц)
2. Система парсит документ и извлекает 12 технических параметров (сечения кабелей, токи, напряжения, типы защитных устройств)
3. Агент обращается к базе нормативов (ПУЭ, ГОСТ Р 50571) и находит применимые требования
4. Система выявляет 3 несоответствия (занижено сечение кабеля, отсутствует УЗО на линии ванной, превышен допустимый ток)
5. Формируется отчёт с указанием: параметр → значение в документе → требование норматива → пункт норматива → рекомендация

### Сценарий: документ без нарушений

1. Пользователь загружает документ, полностью соответствующий нормативам
2. Система извлекает параметры, сверяет с базой
3. Отчёт: «Несоответствий не обнаружено», с перечнем проверенных параметров и нормативов

### Edge-кейсы

| Сценарий | Ожидаемое поведение |
|----------|-------------------|
| Документ в сканированном PDF (без OCR-слоя) | Система сообщает, что не может извлечь текст, и предлагает загрузить текстовый PDF |
| Документ на языке, отличном от русского | Система сообщает об ограничении и просит русскоязычный документ |
| Параметр в документе указан в нестандартных единицах | Агент пытается конвертировать; если не уверен — помечает параметр как «требует ручной проверки» |
| Нужный норматив отсутствует в базе | Система сообщает, что не нашла применимый норматив для конкретного параметра |
| Документ > 50 страниц | Система обрабатывает постранично с предупреждением об увеличенном времени |
| Неоднозначный параметр (диапазон значений) | Агент проверяет обе границы диапазона и отмечает неоднозначность |
| Конфликтующие нормативы (разные требования к одному параметру) | Система выводит оба требования и отмечает конфликт для ручного разрешения |

## 4. Ограничения

### Технические ограничения (SLO)

| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| p95 latency | < 180 с | Пользователь готов ждать до 3 мин за автоматическую проверку |
| Max размер документа | 50 страниц | Ограничение на время обработки и стоимость |
| Max размер файла | 20 MB | Ограничение на upload и парсинг |
| Контекстное окно LLM | 128K tokens | Определяет max chunk size при RAG |
| Rate limits API | ~60 RPM | Ограничение провайдера LLM |

### Операционные ограничения

| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| Бюджет на LLM API (PoC) | < $50/мес | Ограничение на стадии прототипа |
| Количество нормативов в базе | 3–5 документов | Ограничение на PoC-масштаб |
| Поддерживаемые форматы | PDF, DOCX | Минимальный набор для демо |
| Языки документов | Русский | Основной язык технической документации |

## 5. Архитектурный набросок

### Модули системы

```
┌─────────────────────────────────────────────────────────┐
│                    Web UI (Streamlit)                    │
│              Загрузка документов, отчёты                 │
└──────────────────────┬──────────────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────────────┐
│                  Orchestrator Agent                      │
│         Управление пайплайном, state & memory           │
│         Маршрутизация между модулями                    │
└───┬──────────┬───────────────┬──────────────┬───────────┘
    │          │               │              │
┌───▼───┐ ┌───▼────────┐ ┌───▼───────┐ ┌────▼──────────┐
│Parser │ │ Parameter  │ │ Normative │ │    Report     │
│Module │ │ Extractor  │ │ Checker   │ │   Generator   │
│       │ │   (LLM)    │ │  (LLM +   │ │    (LLM)      │
│PDF/   │ │            │ │   RAG)    │ │               │
│DOCX → │ │ Текст →    │ │ Параметры │ │ Несоответствия│
│текст  │ │ параметры  │ │ vs нормы  │ │ → отчёт       │
└───────┘ └────────────┘ └─────┬─────┘ └───────────────┘
                               │
                    ┌──────────▼──────────┐
                    │  Vector Store (FAISS)│
                    │  Нормативная база    │
                    │  (ГОСТы, СП, ПУЭ)   │
                    └─────────────────────┘
```

### Ключевые интеграции

| Компонент | Технология | Назначение |
|-----------|-----------|------------|
| LLM | OpenAI GPT / Anthropic Claude | Извлечение параметров, сопоставление, генерация отчёта |
| Vector Store | FAISS / ChromaDB | Хранение и поиск по нормативной базе |
| Document Parser | PyMuPDF + python-docx | Извлечение текста из PDF/DOCX |
| Embedding Model | OpenAI text-embedding | Векторизация нормативов и запросов |
| Web UI | Streamlit/Node.js | Интерфейс пользователя |
| Agent Framework | Antropic SDK/ LangGraph / CrewAI | Оркестрация агентного пайплайна |
| State Storage | SQLite / In-memory | Хранение промежуточного состояния проверки |

## 6. Data Flow

```
Пользователь                    Система
     │                              │
     │  1. Загружает PDF/DOCX       │
     │─────────────────────────────►│
     │                              │
     │                   ┌──────────▼──────────┐
     │                   │  Parser Module       │
     │                   │  PDF/DOCX → текст    │
     │                   │  [НЕ LLM]            │
     │                   └──────────┬──────────┘
     │                              │
     │                   ┌──────────▼──────────┐
     │                   │  Parameter Extractor │
     │                   │  Текст → JSON        │
     │                   │  [LLM — structured   │
     │                   │   output]             │
     │                   └──────────┬──────────┘
     │                              │
     │                   ┌──────────▼──────────┐
     │                   │  Normative Checker   │
     │                   │  Для каждого парам.: │
     │                   │  1. RAG-поиск нормы  │
     │                   │     [НЕ LLM]         │
     │                   │  2. Сравнение с      │
     │                   │     требованием       │
     │                   │     [LLM — agent]     │
     │                   │  3. Вердикт          │
     │                   └──────────┬──────────┘
     │                              │
     │                   ┌──────────▼──────────┐
     │                   │  Report Generator    │
     │                   │  JSON → отчёт        │
     │                   │  [LLM — generation]  │
     │                   └──────────┬──────────┘
     │                              │
     │  5. Получает отчёт           │
     │◄─────────────────────────────│
```

### Что делегировано LLM/Agent

| Этап | LLM/Agent | Детерминированный код |
|------|-----------|----------------------|
| Парсинг документа | — | PyMuPDF, python-docx |
| Извлечение параметров | LLM (structured output) | Валидация JSON-схемы |
| Embedding нормативов | Embedding model | — |
| Поиск релевантных норм | — | Vector similarity search |
| Сопоставление параметра с нормой | LLM (agent с reasoning) | — |
| Принятие решения о соответствии | LLM (agent) | Пороговая логика для числовых значений |
| Генерация отчёта | LLM (generation) | Шаблон отчёта |
| Загрузка/сохранение файлов | — | Детерминированный код |
| Управление пайплайном | Agent (orchestrator) | Fallback и retry-логика |
